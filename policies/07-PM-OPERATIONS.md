# PM 운영 정책서 (권장 기본 흐름)

이 문서는 강사가 각 기능 에이전트에 직접 붙여넣지 않고,
`Hermes PM(Project Manager) 에이전트` 1명에게만 지시하는 운영 방식입니다.

## 1. 왜 PM 단일 지시가 기본인가

- 강의 상황에서 우선순위가 자주 바뀌므로 `cron`보다 수동 점검이 정확합니다.
- 지시 채널이 1개라서 누락/중복 지시가 줄어듭니다.
- PM이 Linear 상태와 `.agent` 문서를 함께 관리해 추적성이 좋아집니다.

## 2. 강사가 PM에게 보내는 시작 지시문 (복붙)

```text
너는 Argus 프로젝트의 Project Manager 역할이다.
policies/06-LINEAR-TICKET-BACKLOG.md를 기준으로 T1~T11 Linear 티켓을 생성/정리해라.
이미 있는 티켓은 중복 생성하지 말고 업데이트해라.
각 티켓에 제목/설명/완료기준(AC)/라벨/의존성을 정확히 반영해라.
의존성 미충족 티켓은 dep:blocked 라벨을 적용해라.
완료 후 티켓 키 목록, 의존성 맵, 현재 blocked 목록을 보고해라.
```

## 3. 강사가 PM에게 보내는 주기 점검 지시문 (복붙)

```text
현재 남은 작업과 blocked 티켓을 점검해라.
각 티켓 상태를 Doing/Blocked/Ready로 분류해 보고해라.
다음 1시간 내 착수할 작업 3개를 우선순위 순으로 제안해라.
필요하면 담당 에이전트에게 전달할 단일 지시문도 같이 작성해라.
```

## 4. 강사가 PM에게 보내는 종료 점검 지시문 (복붙)

```text
오늘 작업 종료 점검을 수행해라.
완료된 티켓, 미완료 티켓, blocked 사유를 요약해라.
내일 시작 시 첫 작업 3개를 제안해라.
.agent 문서 누락이 있으면 파일명 기준으로 지적해라.
```

## 5. cron 사용 기준

- 기본: 비활성 또는 최소화
- 예외: 하루 1회 요약 알림만 허용
- 권장 대체: 강사가 필요할 때 PM 점검 지시문 수동 실행

## 6. 직접 지시가 필요한 예외

아래 상황에서만 `02~04` 문서를 사용해 기능 에이전트에 직접 지시합니다.

- PM 에이전트 장애/응답 지연
- 특정 역할만 긴급 핫픽스 필요
- 강의 시연상 역할별 프롬프트를 직접 보여줘야 하는 경우
